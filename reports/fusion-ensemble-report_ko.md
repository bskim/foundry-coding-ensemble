# 선택 기반 앙상블 vs 단일 모델 (SWE-bench Verified)

영어 원문은 여기에서 볼 수 있습니다: [fusion-ensemble-report.md](fusion-ensemble-report.md)

이것은 개념 증명(proof of concept)입니다. OpenRouter의 Fusion(여러 모델을
deep-research 답변으로 합성)에서 영감을 받았지만, 답변 합성(synthesis)을 N개 후보 중
하나를 고르는 더 단순한 선택(Select-on-N)으로 바꿔 에이전트 코딩에 맞게 변형했습니다.
그렇게 단순화했음에도 이 앙상블은 테스트 서브셋에서 best-of-N 상한(ceiling)에 도달했고,
이 실험에서 단일 모델 SOTA 베이스라인보다 높은 문제해결 커버리지를 낸 유일한 구성입니다. 아래
수치는 소규모 신호일 뿐 벤치마크급 결과가 아닙니다. 비용과 품질 양쪽 모두 더 발전시킬 여지가 분명히 있습니다
("개선 여지" 절 참고).

서브셋 (5개 인스턴스): matplotlib__matplotlib-24970, mwaskom__seaborn-3187,
pallets__flask-5014, psf__requests-1921, pydata__xarray-7393.

비용 기준은 gross입니다(모든 프롬프트 토큰은 입력 요율로, 모든 완성 토큰은 출력 요율로 계산하며
캐시 할인은 없음). 모든 구성에 같은 방식을 적용하므로 $/해결을 그대로 비교할 수 있습니다. 가격은
[harness/pricing.json](../harness/pricing.json)에 있습니다.

gpt-5.4는 단일 모델 SOTA 베이스라인이며 앙상블 멤버가 아닙니다. 오픈웨이트 패널은
kimi-k2, deepseek-v4-pro, minimax-m2.5, grok-4.3, glm-5.1입니다.

### 오라클 best-of-N 상한

완벽한 선택기가 오픈웨이트 단일 모델 후보 중 항상 해결하는 후보를 골랐다면, 해결 집합은
후보들의 합집합이 됩니다:

- kimi와 deepseek의 best-of-2가 matplotlib, flask, requests, xarray를 커버 = 4/5 (80%).
- 이 서브셋의 상한은 4/5입니다. seaborn-3187은 모든 모델과 구성에서 미해결입니다.

그 4/5 상한은(gpt-5.4 2/5, 최고 단일 모델 3/5, 매 턴 fusion 1~2/5과 비교해) 개선된 접근의
출발점이 되었습니다: 다양성은 유지하되 매 턴 합성은 버리는 것입니다.

## 3. 매 턴 fusion이 실패한 이유와 바꾼 것

- OpenRouter Fusion ([블로그](https://openrouter.ai/blog/announcements/fusion-beats-frontier/))은
  에이전트 코딩이 아니라 deep research(텍스트 답변, 장기 과제 없음)를 위해 설계되었습니다.
  자체 FAQ에서도 Fusion은 코딩 모델의 드롭인 대체재가 아니라고 밝힙니다. 코딩 루프 전반이
  아니라, 어려운 하위 질문에만 선택적으로 부르는 보조 도구에 가깝습니다.
- Fusion이 보고한 이득의 약 3/4는 합성 단계에서, 약 1/4는 모델 다양성에서 나옵니다.
- Mixture-of-Agents ([arXiv 2406.04692](https://arxiv.org/abs/2406.04692))는 계층형
  제안자와 집계기를 사용하며 채팅 및 instruction 벤치마크에서 강하지만, 에이전트 코딩에서는
  평가되지 않았습니다.
- 코드는 검증할 수 있으므로, 합성을 코딩에 옮기면 선택 기반 best-of-N이 됩니다: 다양한 후보를
  만든 뒤 심판(또는 테스트 기반 선택기)으로 하나를 고릅니다. LLM이 파일 전체를 병합하는
  단계가 없는데, 바로 그 병합이 앞서 본 read 루프와 게이트웨이 오류를 일으킨 실패 원인입니다.

### 개선된 설계

1. 패널: K개의 다양한 오픈웨이트 단일 에이전트(agent_eval.py 재사용)가 각각 일반 에이전트
   루프로 후보 패치 하나를 생성합니다. 매 턴 fan-out 없음.
2. 선택기: 오픈웨이트 심판 하나가 이슈와 모든 후보 diff에서 가장 좋은 후보 하나를 고릅니다.
   이는 합성이 아니라 선택입니다.
3. 선택된 패치를 동일한 Docker 하니스로 채점해 비교 가능한 해결률과 $/해결을 얻습니다.

토큰 비용은 대략 단일 실행 K번에 작은 심판 호출 하나를 더한 정도여서, 매 턴 fan-out의 6~7백만
토큰보다 훨씬 낮습니다.

## 4. 개선된 결과: Plan A (선택)과 Plan B (합성)

에이전트 루프, 30턴, Docker 하니스, run_id refine. 4개의 오픈웨이트 단일 에이전트
(kimi-k2, deepseek-v4-pro, minimax-m2.5, glm-5.1) 패널이 각각 후보 패치 하나를 생성합니다.
같은 후보 집합을 두고 두 집계기를 실행합니다:

- Plan A, fusion-select: 오픈웨이트 심판(deepseek-v4-pro)이 인스턴스마다 가장 좋은 후보
  패치 하나를 고릅니다. 선택일 뿐 병합은 없습니다.
- Plan B, fusion-synth: deepseek-v4-pro가 마지막에 한 번, 모든 후보 패치를 하나의
  unified diff로 합성합니다. 실행 단계의 apply-gate가 적용되어, 합성된 diff는 인스턴스
  컨테이너에서 `git apply --check`를 통과해야 하고, 통과하지 못하면 fusion-select 패치로
  되돌립니다(fallback).

### 4.4 발견

1. 품질이 상한에 도달. 두 집계기 모두 4/5 (80%), 즉 오라클 best-of-N 상한에 100% 선택기
   효율로 도달합니다. 이는 여기 모든 구성보다 문제해결 커버리지가 높습니다: gpt-5.4 2/5, 최고 단일 모델 3/5,
   매 턴 fusion 1~2/5. 어떤 단일 오픈웨이트 모델도 3/5를 넘지 못하지만, 다양한 패널의
   합집합은 4/5에 도달하고, 심판(선택)과 합성기(합성) 모두 그 전체 합집합을 회수합니다.
2. 토큰은 줄었지만 공짜는 아닙니다. 매 턴 fusion의 6~7백만 대비 5.53백만 토큰인데, 이제
   비용이 매 턴 fan-out이 아니라 실제 에이전트 후보 K개(4 곱하기 약 1.3백만)에서 나오기
   때문입니다. 집계 호출 자체는 무시할 만한 수준으로, synth는 select 대비 약 722 토큰
   (약 0.003달러)만 더 씁니다.
3. $/해결은 여전히 최고 단일 모델보다 높습니다. 해결당 1.79는 solo-minimax의 0.17의 약 10배인데,
   추가 해결(오직 deepseek만 푸는 xarray) 하나를 얻으려고 후보 4개를 생성하는 비용을 치르기
   때문입니다. 개선된 앙상블은 비용을 더 높은 품질 상한과 맞바꿉니다. 비용은 더 들면서
   품질까지 잃었던 매 턴 fusion과는 정반대의 트레이드오프입니다.
4. 원자적 수정에서는 선택과 합성이 거의 같습니다. 둘 다 같은 인스턴스에서 4/5에 도달합니다.
   SWE-bench류 수정에서는 해결 가능한 이슈를 최소 한 후보가 통째로 풀기 때문에 합칠 것이
   없습니다: 합성이 선택을 넘어설 수 없고, apply-gate는 이득 없이 위험(잘못 병합된
   diff)만 더합니다. 보고된 합성 이득은 부분 답변이 실제로 합쳐지는 deep-research 답변
   합성에서 측정된 것이라, 원자적 코드 패치에는 그대로 적용되지 않습니다. 단순함을 위해
   기본은 fusion-select를 쓰고, 올바른 수정이 여러 후보에 흩어져 있는 과제에만 fusion-synth를
   쓰세요.

## 5. 개선 여지

이것은 작은 개념 증명이며, 핵심 결과(Select-on-N이 best-of-N 상한에 도달)는 5개 인스턴스에서만
확인됐습니다. 더 저렴하고 강하게 만드는 방향은 분명합니다:

- 평가 규모 확대. 5개 인스턴스는 신호일 뿐 벤치마크가 아닙니다. 더 큰 Verified 서브셋
  (50~500개)에서 다시 돌려 상한 효과를 검증하고 안정적인 비용 수치를 확보하세요.
- 패널 가지치기. 비용 최소 패널은 이미 같은 품질에서 비용을 절반으로 줄입니다. 학습하거나
  측정한 상호 보완성 점수를 활용하면, 리포별/이슈 유형별로 상한을 포괄하는 가장 작은 패널을
  고를 수 있습니다.
- 항상 N개를 돌리는 대신 라우팅. 가장 저렴한 단일 모델로 시작하고, 후보들이 엇갈리거나
  테스트가 계속 실패할 때만 패널로 올려, 전체 패널 비용을 어려운 인스턴스에서만 치르세요.
- 선택을 테스트 기반으로. 지금 심판은 LLM입니다. 선택 전에 후보마다 리포 자체 테스트를
  돌리면 선택 근거가 검증 가능한 신호가 되고, 심판의 추론에 대한 의존이 줄어듭니다.
- 상한 자체를 올리기. 지금 상한은 패널 구성에 묶여 있습니다: seaborn-3187은 모든 멤버가
  못 풉니다. 현재 미해결 인스턴스를 푸는 모델을 추가하면, 선택이 회수할 수 있는 오라클
  상한 자체가 올라갑니다.
- 최신 SOTA 베이스라인과 다시 비교. gpt-5.4 대비 이점은 더 강한 베이스라인이 나올 때마다
  다시 측정해야 합니다.

요점: 단순화된 선택 전용 앙상블조차 이 서브셋에서 이미 단일 모델 SOTA보다 높은 문제해결 커버리지를
내고, 가지치기와 라우팅으로 비용을 상당히 낮출 수 있습니다. 더 큰 규모와 테스트 기반 선택기를
더하면 두 축 모두에서 의미 있는 여유가 있습니다.

## 6. 범용 코딩 선택기로 가는 길

SWE-bench에는 실무 코딩에 없는 두 가지가 있습니다: 고정된 문제 집합과, 각 패치를
채점하는 숨은 gold 테스트입니다. 그래서 이런 의문이 자연스럽게 따라옥니다: 이게 범용적인
접근일까, 아니면 정답이 이미 알려진 벤치마크에만 맞춰진 걸까?

벤치마크에 맞춰진 부분과 그렇지 않은 부분:

- 선택기는 gold 테스트를 본 적이 없습니다. fusion-select는 이슈 테스트만 보고 후보
  diff 중 하나를 고르며, 이는 실제 코딩 에이전트가 가진 정보와 같습니다. gold 테스트
  채점은 선택이 끝난 뒤 오프라인에서 일어납니다. 따라서 선택 메커니즘 자체는 범용입니다.
- 다만 벤치마크에 맞춰져 조심해서 읽어야 할 결과가 하나 있습니다: 4.5의 비용 최소 2-모델
  패널은 인스턴스별 결과를 본 뒤 고른 구성입니다. 이는 사후(post-hoc) oracle 적합이며
  범용 성능의 근거가 아닙니다.

SWE-bench 밖에서는 어떻게 평가할까요. 실무에서는 모든 과제를 gold 테스트로 채점하지
않고, 사람의 정성 평가가 채점을 대신합니다. 여기서 SWE-bench는 방법 검증용 하니스로만
씁니다: 객관적으로 채점되기 때문에, 우리가 확인하고 싶은 단 하나의 질문에 답할 수
있습니다 - 이렇게 구현한 결과물이 단일 모델보다 더 높은 점수를 받는가? 실행 기반 선택이 SWE-bench
점수를 올린다면, 그 메커니즘은 정답을 외운 게 아니라 실제로 제 역할을 하고 있다는 뜻입니다.

### Plan 1: 실행 기반 선택 (런타임, gold 테스트 없음)

LLM 단독 심판 대신, 에이전트가 런타임에 실제로 가지고 있는 검증 가능한 신호들을
점수화해 결합합니다:

1. 빌드/컴파일 통과 (가장 저렴하면서도 확실한 1차 필터; 빌드가 안 되는 후보는 탈락).
2. 기존 테스트 회귀 검사 (현재 통과 중인 테스트를 깨는 후보 감점).
3. 자작 테스트: 각 패널 에이전트가 수정과 그 수정을 검증하는 테스트를 함께 작성하고, 그
   테스트를 다른 후보들에 교차 적용해 런타임 oracle로 씁니다.
4. 타입 체크와 린트 (mypy / tsc / ruff 등).
5. 후보 합의 (독립 에이전트들이 비슷한 diff로 수렴하면 신뢰 가산).
6. LLM 심판은 실행 신호가 동점일 때만 타이브레이커로 사용.

선택 점수 = f(빌드, 회귀, 자작 테스트 통과율, 타입/린트, 합의, 심판). 이렇게 하면 선택이
추측에서 검증 가능한 결정으로 바뀌고, 그래서 SWE-bench 밖에서도 작동합니다.

### Plan 2: 범용 평가 방법론

- Held-out 테스트 분리: repo 테스트를 에이전트와 선택기에 보이는 세트와 채점에만 쓰는 숨은
  세트로 나눕니다. 이는 실제 개발 방식을 그대로 본떠, 선택기가 정답에 닿지 않게 합니다.
- 규모와 다양성: 50~500개 인스턴스, 두 개 이상의 언어와 작업 유형(파이썬 버그픽스만이
  아니라)으로 넓혀 패널과 선택기가 과제 분포에 robust한지 확인합니다.
- Train/test 패널 분리: 어떤 모델을 패널에 넣을지는 train split에서 정하고, 수치는 test
  split에서만 보고합니다. 2-모델 패널 뒤의 사후 적합을 제거합니다.
- 테스트 없는 과제(문서, UI, 리팩터): LLM 루브릭 심판을 사람 평가와 캘리브레이션하거나,
  프록시 신호(CI 통과율, 리뷰 수락율, 머지 후 revert율)를 장기 근거로 수집합니다.

### Plan 3: 비용 인지 라우팅 (Plan 1 기반)

모든 과제에 N개 에이전트를 돌리는 건 실무에서 비현실적입니다.

- 난이도 라우팅: 가장 싼 단일 모델을 먼저 시도하고, 빌드와 테스트가 통과하면 종료.
- 불일치 시 에스컬레이션: 1차가 실패하거나 후보들이 엇갈릴 때만 패널 호출.
- 조기 종료: 한 후보가 가용한 테스트를 모두 통과하면(Plan 1 신호) 나머지 평가 생략.

### 측정 프로토콜

Plan 3은 Plan 1 위에 구현 가능하므로 둘을 순서대로 측정합니다:

1. Plan 1을 SWE-bench에서 측정: 실행 기반 선택이 LLM 심판 버전이 도달한 커버리지를
   유지하는가?
2. Plan 1 + 3을 측정: 라우팅이 같은 커버리지를 유지하면서 $/해결을 낮추는가?

| 단계 | 변경 | 성공 기준 |
|------|------|----------|
| 1 | 선택기: LLM 단독 -> 실행 기반 | gold 테스트 없는 선택이 oracle 선택 효율을 유지 |
| 2 | held-out 테스트 분리, 50~500으로 확대 | 작은 subset의 100%에서 선택 효율이 무너지지 않음 |
| 3 | train/test 분리로 패널 선택 | 사후 적합 없이 단일 모델 대비 커버리지 우위 유지 |
| 4 | 다언어, 다작업 유형 | 분포 밖에서도 커버리지 이득 재현 |
| 5 | 난이도 라우팅 추가 (Plan 1 + 3) | 같은 커버리지에서 $/해결 대폭 감소 |

이 subset 실측 (4-모델 패널: minimax-m2.5, deepseek-v4-pro, kimi-k2, glm-5.1;
gold 테스트 채점, 동점일 때만 심판):

| 구성 | 해결 | $/해결 | 토큰 |
|------|------|--------|------|
| Plan 1 (실행 기반 선택) | 4/5 (80%) | 1.79 | 5.53M |
| Plan 1 + 3 (비용 인지 라우팅) | 4/5 (80%) | 0.29 | 1.66M |

라우팅은 같은 80% 커버리지(동일한 네 인스턴스 해결, seaborn-3187은 둘 다 실패)를
유지하면서 토큰을 약 70%, $/해결을 약 84% 줄였습니다. 가장 싼 후보부터 보고 그 후보의
검증 가능한 신호가 깨끗하면 거기서 멈추며, 한 인스턴스에서만 두 번째 후보로
에스컬레이션했기 때문입니다. 실행 기반 Plan 1은 4절의 LLM 심판 선택기와 같은 4/5를
재현했으므로, gold 테스트 없는 신호로 바꿔도 여기서는 커버리지 손실이 없었습니다. 다섯
인스턴스라 벤치마크급이 아니라 작은 신호입니다.

이것은 여전히 개념 증명이고, 내세우는 주장도 제한적입니다. 선택 메커니즘은 정답을
읽지 않기 때문에 범용적이며, 이를 실제 코딩 작업에서 입증하려면 gold 테스트 oracle을
런타임 검증 신호로 대체하고 held-out + train/test 분리 방법론으로 다시 검증해야 합니다.
